# Lessons Log

Record project-specific gotchas, patterns, and knowledge here. Future sessions will read this to avoid repeating mistakes.

| Lesson | Context |
|--------|---------|
| When updating role files to reference a new artifact, also update the corresponding `/start` skill's context loading phase. Role files and start skills must stay in sync. | QA review found PM, QA, DevOps role.md updated to consume architecture.md but their /start skills weren't. |
| When creating a new expert, update CLAUDE.md repo guide in the same task — not as a follow-up. | System Architect was created but CLAUDE.md repo structure wasn't updated. |
| When a script's directory depth changes during a restructure, use a walk-up approach (search for a known marker directory like `experts/`) instead of hardcoding `../..` levels. This makes the script work regardless of where it's placed — in the repo or in a built package. | `install-team.sh` moved from 2 to 3 levels deep but needs to work at both depths since package.sh copies it to a different location. |
| When a documentation file (like SKILL.md) is also a runtime artifact that gets packaged, updating its path references requires updating the packaging scripts in lockstep. Treat documentation-and-packaging as an atomic unit — you can't update one without the other. | SKILL.md references `$SKILL_DIR/...` paths that must match the actual file layout inside the `.skill` package. Updating SKILL.md without updating `package.sh` would break the deployed skill. |
| When adding a new expert, update ALL reference tables that list experts — not just generated configs. Per-target READMEs with prefix tables, SKILL.md expert lists, and similar enumerations are easy to miss. | System Architect was added to install.sh and generated configs, but `targets/ide/cursor/README.md` and `targets/ide/claude-code/README.md` prefix tables were not updated. |
| Don't reference documentation files that don't exist yet. Either create them in the same task or don't link to them. Broken links compound — each referencing file becomes a separate fix later. | `overview.md`, `getting-started.md`, and `workflow-anatomy.md` are referenced in 4+ files but never created. |
| When adding a new artifact to `/start` skills, also add it to the role.md Session Protocol. The `/start` skill governs issue execution, but the Session Protocol governs all other skill entry points. They must stay in sync. | `architecture.md` was added to all `/start` skills (swe-bug-007 fix) but not to QA, DevOps, and PM role.md Session Protocols. Non-`/start` skills like `/review` or `/pipeline` would miss the architecture context. |
| Budget for QA-driven rework in decomposition. M3-M7 planned 11 issues but delivered 28 — the cleanup tail was 2.5x the original plan. QA reviews consistently find cross-expert consistency issues that require follow-up SWE sessions. | Postmortem of M3-M7. Both QA reviews found 5 issues each. All were resolvable in 1-2 sessions, but they weren't anticipated in the original decomposition. |
| Three of seven lessons are about keeping references in sync across experts. Consider a "when you modify an expert" checklist: role.md, /start skill, Session Protocol, docs-protocol, agent-reference, install scripts, target READMEs, SKILL.md. | Postmortem pattern analysis. The same class of bug (update one artifact, miss the parallel one) recurred across both milestones. |
| For small, low-risk milestones (template-only changes, no cross-expert interactions), skipping QA is the right call. The QA rework budget lesson from M3-M7 applies to milestones that touch cross-expert boundaries — not to isolated single-expert changes. | M8 postmortem. 2 planned, 2 delivered, 0 rework. First milestone with no QA gap. |
| Dogfooding the workflow during the session that changes it is effective validation. The `/add-feature` interview that scoped M8 demonstrated the exact friction that M8 was designed to fix. | M8 postmortem. The PM interview ran all 5 categories for a clearly small feature, confirming the need for the adaptive interview. |
| When a change applies uniformly across multiple experts, scope it as one task rather than splitting per expert. The work is mechanical and repetitive — splitting creates planning overhead without reducing execution risk. | M9 postmortem. swe-feature-031 touched 15 files across 4 experts in one session. |
| Consider scoping related changes as a single milestone instead of sequentially. M8 (PM date removal) and M9 (all-expert date removal) could have been one milestone, saving a planning cycle. The incremental approach validated the pattern first, but the trade-off was marginal for low-risk work. | M9 postmortem. M8+M9 = 3 sessions, 0 rework. |
| Cursor `alwaysApply: true` workspace rules are loaded on every message regardless of session context. When multiple expert roles are defined as always-applied rules, every session gets all roles — only one of which is relevant. Prefer conditional loading for role-specific content. | M10 context optimization research. 5 expert roles (~350 lines) always loaded; only 1 (~70 lines) is needed per session. |
| Session protocols (in role files) and `/start` skills (in skill files) can silently diverge on what context to load. When updating one, check the other. They serve the same purpose (session startup) but are maintained independently. | M10 research found 4 inconsistencies: QA missing own handoff, DevOps missing SWE handoff, SA misaligned on lessons-log and cross-expert handoffs, PM over-broad on handoff loading. |
| When designing for multiple platforms with different capability sets, find the lowest common denominator for core behavior and add platform-specific enhancements as bonuses, not requirements. This keeps behavior consistent and avoids maintaining divergent mechanisms. | M11 architecture design: Cursor lacks session-end hooks but Claude Code has them. Rather than using hooks on Claude Code and a different mechanism on Cursor, both platforms use the same soft-trigger approach for handoff. Claude Code's SessionStart hook is a bonus, not a dependency. |
| SKILL.md `description` fields are the primary discovery mechanism for agent-decided skills. They should answer two questions: "What does this skill do?" and "When should the agent use it?" — not just the first. A description like "Generate project brief" is less effective than "Generate the project brief from interview notes. Use when the user has completed a project interview." | M11 architecture design: Cursor skill discovery reliability (~50%) depends heavily on description quality. Descriptions that include trigger conditions perform better than those that only describe output. |
